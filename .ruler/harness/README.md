# Harness Engineering — 다중 에이전트 협업 모델

> 하네스(Harness)는 LLM 엔진을 "실제로 쓸 수 있는 제품"으로 만들어주는 실행 골격이다.
> 이 폴더의 규칙은 AI 코딩 에이전트가 **단독으로 코드를 쏟아내지 않고**,
> 역할별로 나뉘어 협업하도록 강제한다.

---

## 👶 처음이면 여기부터 (For Dummies)

### 이게 대체 뭔데요?

AI한테 "이거 만들어줘" 한 마디로 시키면, 처음엔 잘 해주는 것 같다가 **길어질수록**:
- 엉뚱한 걸 만들고 있고 (드리프트)
- 요청한 적 없는 것도 잔뜩 붙이고 (과잉 구현)
- 이미 있는 코드를 또 만들고 (중복)
- 유저는 뭐 하는 중인지 모르고 (블랙박스)
- 버그·보안 구멍을 놓칩니다 (단일 관점)

**하네스는 AI 한 명한테 8명 역할을 번갈아 시키는 방법입니다.**
"지금은 기획자(Planner) 역할이야 → 이제 구현자(Implementer) → 이제 리뷰어..." 이런 식으로요.

### 왜 그게 도움이 되나요?

사람이 혼자 기획·구현·검토를 한꺼번에 하면 **뭐가 놓였는지 본인이 모릅니다.**
역할을 바꿔가며 다시 보면 보여요. AI도 똑같습니다. 같은 LLM이어도 "지금 나는 리뷰어"라는 프레임을 갖고 보면 구현자 모드에서 못 본 게 보입니다.

### 내가 해야 할 건 단 3개

1. **`vision.md` 에 뭘 만들고 싶은지 채우기** — 한 줄 목표만 있어도 시작 가능
2. **AI 툴 열면 자동 로드됨** — 복붙 필요 없음 (AI-Ruler 앱에서 툴을 선택했다면)
3. **`[Reporter]` 로 시작하는 1줄 보고를 읽고** — 이상하면 개입, 괜찮으면 "진행해"

그게 전부입니다. 8개 에이전트, 핸드오프, DRIFT, VIOLATION... 처음부터 다 알 필요 **없어요.** AI가 해요. 당신은 **감독**입니다.

### 🪄 자동 부트스트랩 — 스텝바이스텝 (For Dummies)

AI-Ruler 앱에서 "사용 중인 AI 툴" 을 선택하고 ZIP을 받았다면, 매 세션 프롬프트 복붙이 **필요 없습니다.**
AI 툴들은 **프로젝트 루트의 특정 파일**을 자동으로 읽는 규약이 있어요. ZIP이 그 파일을 만들어줬습니다.

#### 예시 — Claude Code 유저

**1. ZIP 풀기**

다운받은 ZIP을 프로젝트 루트에 압축 해제. 이렇게 생깁니다:

```
my-cafe-app/
├── CLAUDE.md              ← ★ 이게 핵심. Claude Code가 자동으로 읽음
├── vision.md              ← 당신이 채울 파일
├── base.md, backend.md … ← 기본 코딩 규칙
├── harness/
│   ├── README.md, workflow.md, walkthrough.md
│   └── agents/01-planner.md ~ 08-reporter.md
├── src/
└── package.json
```

**2. vision.md 채우기** (5분)

§1, §3, §4, §5 네 개만 채우면 OK. (아래 예시 참고)

**3. Claude Code 열기**

터미널에서 `claude` 명령어 실행. 그리고 그냥 평범하게 말 걸기:

```
유저: users 리스트 엔드포인트 만들어줘
```

**4. Claude가 이렇게 반응합니다** (자동으로!)

```
[Reporter] CLAUDE.md 로드됨 · vision.md와 harness/ 확인
[Planner] 요청: users 리스트 엔드포인트
  vision.md §4 in-scope 확인 중...
  Sub-Goal #1 초안 작성 · Researcher 질의 대기
[Researcher] 기존 users 테이블 스키마 조사 중...
...
```

**그게 끝.** 프롬프트 복붙 **안 했어요.** `CLAUDE.md` 가 자동 로드되어서 Claude가 스스로 8 에이전트 모드로 작동한 겁니다.

#### 예시 — Cursor 유저

**1. ZIP 풀기** (위와 동일)

ZIP엔 `.cursor/rules/harness.mdc` 가 들어있음. Cursor가 자동 인식.

**2. Cursor 열기**

```
my-cafe-app/
├── .cursor/rules/harness.mdc  ← ★ Cursor가 자동 로드
├── vision.md
├── harness/ …
└── src/
```

Cursor Settings → **Rules**에서 `harness.mdc` 가 `alwaysApply` 로 표시되는지 확인.

**3. 평범하게 질문**

```
유저: 재고 아이템 생성 기능 추가
```

Cursor가 자동으로 Planner 역할부터 시작합니다.

#### 예시 — GitHub Copilot 유저

ZIP엔 `.github/copilot-instructions.md` 포함. Copilot 최신 버전이 자동으로 레포 지침으로 인식.

#### 예시 — 기타 툴 / 직접 설정

ZIP엔 `HARNESS-BOOTSTRAP.md` 포함. 그 파일 안에 **복붙용 프롬프트 + 툴별 경로 안내**가 다 들어있음.

#### 진짜 자동 로드 됐는지 확인하는 방법

새 세션 열고 아무 요청 없이 딱 한 줄 물어보세요:

```
유저: 지금 너 어떤 역할이야?
```

- ✅ **정상**: `"Planner 역할로 대기 중입니다. vision.md를 읽었고..."` 같은 답
- ❌ **실패**: `"저는 AI 어시스턴트입니다"` 같은 일반 답 → 자동 로드 안 됨, 수동 프롬프트 필요

#### 자동 로드 실패했을 때 (당황하지 말고)

1. ZIP을 **프로젝트 루트**에 풀었는지 확인 (하위 폴더 안 됨)
2. 해당 파일이 존재하는지 확인 (`CLAUDE.md`, `.cursor/rules/harness.mdc` 등)
3. AI 툴을 완전히 재시작
4. 그래도 안 되면 `HARNESS-BOOTSTRAP.md` 열어서 거기 있는 프롬프트를 한 번 복붙 → 이번 세션만 수동으로

### 오늘 안에 시작하는 5분 가이드

**1단계 — vision.md 열기 (2분)**

프로젝트 루트의 `vision.md` 를 열어서 다음 4개 섹션만 먼저 채우세요:

- **§1 한 줄 목표** — "뭘 만들려는지" 기술 용어 없이 한 문장
- **§3 성공 기준** — 숫자로 측정 가능한 기준 2~4개
- **§4 In Scope** — 할 것 목록
- **§5 Out of Scope** — **절대** 하지 말 것 목록 (이게 가장 중요!)

나머지 섹션은 나중에 채워도 됩니다.

**2단계 — AI 에이전트 열기 (10초)**

Claude Code, Cursor Chat, Copilot Chat 뭐든 OK.

**3단계 — 부트스트랩 파일이 자동 로드되는지 확인 (10초)**

AI-Ruler 앱에서 **사용 중인 AI 툴을 선택했다면** 이미 자동 로드 파일이 ZIP에 포함되어 있습니다.
ZIP을 프로젝트 루트에 풀면 해당 툴이 다음 파일을 알아서 읽습니다 → **프롬프트 복붙 불필요**.

| AI 툴 | 자동 로드 파일 | 확인 방법 |
|-------|---------------|----------|
| Claude Code | `CLAUDE.md` (루트) | 새 세션 시작 시 자동 |
| Cursor | `.cursor/rules/harness.mdc` | Settings → Rules에서 활성 확인 |
| GitHub Copilot | `.github/copilot-instructions.md` | 자동 (Copilot 최신 버전) |
| 직접 설정 / 기타 | `HARNESS-BOOTSTRAP.md` | 매 세션 프롬프트 복붙 |

**3-1단계 (대안) — 자동 로드 미지원 시 이 프롬프트 복붙 (20초)**

```
이 프로젝트는 하네스 엔지니어링 룰셋을 사용해.
다음 순서로 작동해줘:

1. /vision.md 와 /harness/README.md 를 읽어.
2. /harness/workflow.md 의 핸드오프 규약을 따라.
3. Planner 역할로 시작해서 sub-goal 목록을 만들어.
4. 역할 전환 시 해당 /harness/agents/*.md 파일을 다시 읽어.
5. 모든 핸드오프마다 Guardian 판정을 하고, Reporter는 1줄로 보고해.
```

**4단계 — 보고 읽기 (계속 반복)**

AI가 이렇게 1줄 보고를 보냅니다:

```
[Planner] vision.md §4에서 sub-goal 3개 생성 · Researcher 질의 1건
[Researcher] fastify DTO 검증 조사 완료 (옵션 A/B) · Planner 복귀
[Implementer] Sub-Goal #1 구현 완료 · Reviewer 대기
[Guardian] PASS · Security Auditor로 이동
```

- **납득되면** → "계속 진행해"
- **이상하면** → "vision.md §5에 out-of-scope로 적혀있는데?" 같이 근거 제시
- **모르겠으면** → "Reporter, 현재 상태 요약해줘" 라고 요청

**5단계 — Guardian PASS 신호 = 한숨 돌리기**

`[Guardian] PASS` 가 찍히면 그 sub-goal은 vision.md와 정합. 다음 sub-goal로 넘어갑니다.

### 자주 하는 실수 (절대 피하세요)

| 실수 | 왜 망하는가 | 대신 이렇게 |
|------|------------|------------|
| vision.md 안 쓰고 바로 시작 | Guardian이 체크할 기준이 없음 → 드리프트 방치 | 한 줄 목표라도 먼저 |
| 중간에 계속 "이것도 추가" | 스코프가 무한 팽창 | vision.md §4/§5 갱신 후 Planner 재호출 |
| Reporter 보고 안 읽음 | 엉뚱한 방향 가는 걸 놓침 | 1줄만 훑어도 충분 |
| 에이전트 순서 뒤집기 ("리뷰 건너뛰어") | 검증 공백 | workflow.md 순서 유지 |
| 모든 작업에 하네스 적용 | 짧은 작업엔 과하다 | 아래 "필요/불필요" 참고 |

### 하네스가 **필요 없는** 경우

**짧고 단순한 작업엔 오히려 비효율입니다.**

- 30분 내 끝나는 리팩토링
- 혼자 쓰는 1회성 스크립트
- 실험·프로토타입 (버려도 되는 코드)
- "이 변수명만 바꿔줘" 수준의 마이크로 작업

이런 건 기존 `.ruler/*.md` 만으로 충분. 하네스는 켰다 껐다 할 수 있으니 부담 없이.

### 하네스가 **진짜 빛나는** 순간

- **팀 프로젝트** — 리뷰 프로세스를 AI 단계에서 강제
- **프로덕션 서비스** — 보안·테스트 누락이 치명적
- **긴 프로젝트** — 여러 날·여러 세션 이어갈 때 드리프트 방지
- **AI에게 자율성은 주고 싶지만 통제도 유지**하고 싶을 때

### 겁먹지 마세요

- **처음 한 번은 무조건 어색합니다.** 3~4 sub-goal 굴려보면 리듬이 잡혀요.
- **AI가 역할을 잊으면** "지금 너는 <에이전트명> 역할이야, 해당 파일을 다시 읽어" 한 마디면 복구됩니다.
- **너무 형식적이다 싶으면** vision.md § 번호만 지키고 나머지는 느슨하게 가도 돼요. 핵심은 **vision.md ↔ Guardian ↔ Reporter 3개 축**입니다.

### 한 줄 요약

> **"AI가 뭘 하고 있는지 계속 보이고, 벗어나면 멈추고, 끝나기 전에 여러 번 검토받게 만드는 장치."**
>
> 그 이상도 그 이하도 아닙니다.

## 왜 하네스인가

단일 에이전트 실패 패턴:

1. **드리프트(drift)** — 대화가 길어질수록 원래 목표에서 이탈한다.
2. **과잉 엔지니어링** — 요청한 적 없는 추상화·방어 코드를 쌓는다.
3. **블라인드 구현** — 기존 코드·유틸을 모른 채 중복 구현한다.
4. **블랙박스 진행** — 유저가 "지금 뭐 하고 있는지" 모른다.
5. **단일 검토 관점** — 기능만 보거나 구조만 보거나 보안만 본다.

하네스는 **역할 분리**로 이 다섯을 전부 막는다.

## 전체 흐름

```
[유저: vision.md 작성·갱신]
        │
        ▼
 ┌──────────────┐    질문    ┌──────────────┐
 │  Planner     │ ◄────────► │  Researcher  │
 │ (기획자)      │   응답     │  (리서처)     │
 └──────┬───────┘            └──────────────┘
        │ sub-goal 목록
        ▼
 ┌──────────────┐
 │  Implementer │  ← Guardian 수시 정합성 체크
 │ (코드작성자)  │
 └──────┬───────┘
        │ diff
        ▼
 ┌──────────────┐
 │   Reviewer   │ (구조/네이밍/DRY)
 └──────┬───────┘
        ▼
 ┌──────────────────┐
 │ Security Auditor │ (OWASP/시크릿/인증)
 └──────┬───────────┘
        ▼
 ┌──────────────┐
 │      QA      │ (기능/엣지/회귀)
 └──────┬───────┘
        ▼
 ┌──────────────┐   매 단계 ┌──────────────┐
 │   Guardian   │ ◄────────│   Reporter   │ → 유저
 │  (감시자)     │          │   (비서)      │
 └──────────────┘          └──────────────┘
```

## 8개 역할

| # | 역할 | 파일 | 핵심 책임 |
|---|------|------|-----------|
| 1 | Planner (기획자) | `agents/01-planner.md` | vision.md → sub-goal 분해 |
| 2 | Researcher (리서처) | `agents/02-researcher.md` | 기존 코드·패턴 조사 |
| 3 | Implementer (코드작성자) | `agents/03-implementer.md` | sub-goal 1개씩 구현 |
| 4 | Reviewer (리뷰어) | `agents/04-reviewer.md` | 코드 품질 검토 |
| 5 | QA | `agents/05-qa.md` | 기능·엣지·회귀 검증 |
| 6 | Security Auditor (보안검토자) | `agents/06-security-auditor.md` | 보안 전용 검토 |
| 7 | Guardian (감시자) | `agents/07-guardian.md` | vision.md 정합성 감시 |
| 8 | Reporter (비서) | `agents/08-reporter.md` | 유저 실시간 보고 |

## 빠른 시작

1. `/vision.md` 를 열고 템플릿의 빈칸을 채운다 (한 줄 목표, 사용자, 성공 기준 등).
2. AI 에이전트에게 "vision.md를 읽고 Planner 역할로 시작하라"고 지시한다.
3. Planner가 만든 sub-goal 목록을 유저가 확인·승인한다.
4. 이후 핸드오프는 `workflow.md`의 규약을 따라 자동 진행된다.
5. Reporter가 각 단계를 1줄로 보고한다. 유저는 언제든 끊고 vision.md를 갱신할 수 있다.

> **처음이면 먼저 `walkthrough.md` 를 읽어라** — sub-goal 하나가 8명의 에이전트를 통과하는 전 과정을 카페 재고 SaaS 예시로 풀어놓은 실전 데모.

## 실전 구동 방법 (AI 툴별)

대부분의 AI 코딩 에이전트는 "8명을 동시에" 돌리지 못한다 — 하나의 LLM이 **역할 전환**으로 시뮬레이션한다.

### 공통 원칙

- 각 턴에 **하나의 역할만 수행**한다.
- 역할 시작 전 **항상 해당 에이전트 md 파일을 읽는다**.
- 역할 종료 시 **명시적으로 다음 에이전트를 지정**한다 ("이제 Reviewer로 전환합니다").
- 역할 전환 직전 **Guardian 호출** — 프롬프트에 "Guardian 판정 후 다음 단계로"를 루틴화.

### Claude Code / Cursor / Copilot 공통 부트스트랩 프롬프트

```
이 프로젝트는 하네스 엔지니어링 룰셋을 사용한다.
반드시 다음 순서로 작동하라:

1. /vision.md 를 읽는다.
2. /harness/README.md, /harness/workflow.md 를 읽는다.
3. 현재 네 역할을 선택하거나 유저가 지정한 역할을 수행한다.
4. 역할 수행 전 해당 /harness/agents/*.md 를 읽어 규칙을 로드한다.
5. 역할 종료 시 핸드오프 대상과 전달물을 명시한다.
6. 모든 핸드오프 직후 Guardian 판정 1턴을 수행한다.
7. Reporter 출력은 항상 1줄.

작업을 시작한다: Planner 역할로 vision.md를 분해하라.
```

### 서브 에이전트 기능이 있는 툴 (Claude Code Task, Cursor Agents)

- 각 에이전트를 서브태스크로 분리해 병렬 호출 가능
- Guardian·Reporter는 메인 턴에 유지 (판정·보고는 주체가 하나여야 함)

### 역할 전환이 흔들릴 때 (긴 세션)

- 토큰이 누적되면 에이전트가 "자기가 누군지" 잊는다
- 해결: 매 3~5턴마다 Reporter가 "현재 상태 요약" 출력 + 역할 파일 재로드
- 또는: 각 sub-goal 완료 시 새 세션으로 전환, vision.md + 이전 완료 목록만 인계

## 기존 룰셋(.ruler/*.md)과의 관계

하네스 규칙은 **기존 코딩 룰을 대체하지 않는다**. 다음과 같이 계층 구조로 겹친다:

```
vision.md                           ← WHAT (무엇을 만드는가)
  └─ harness/agents/*.md            ← WHO (누가 어느 역할로)
       └─ .ruler/base.md + stack.md ← HOW (어떻게 코드를 쓰는가)
```

- **Planner**: vision.md 분해 시 `.ruler/*.md`의 제약을 sub-goal에 반영 (예: "base.md 네이밍 규칙을 따라야 함").
- **Implementer**: 코드 작성 시 `.ruler/base.md`, `.ruler/frontend.md`/`.ruler/backend.md`를 절대 규칙으로 따른다.
- **Reviewer**: `.ruler/*.md` 위반이 있으면 즉시 BLOCKER.
- **Security Auditor**: `.ruler/security.md`를 최소 기준선으로 쓴다.
- **QA**: `.ruler/testing.md`의 테스트 규칙을 따른다.

하네스는 **협업 구조**, `.ruler/`는 **코드 규칙**. 둘을 겸비하라.

## 토큰 비용 고지

하네스를 켜면 한 작업당 LLM 호출이 평균 4~8배 증가한다.
이유:
- 각 에이전트가 자신의 역할 규칙을 매 호출마다 로드
- 핸드오프 지점마다 Guardian 체크 1회 추가
- Reporter가 매 단계 요약 1회 추가

대신 얻는 것: 드리프트·보안 누락·리뷰 누락이 구조적으로 막힘.
짧은 일회성 작업엔 비효율, **긴 프로젝트·팀 협업·프로덕션 코드에 유리하다.**
